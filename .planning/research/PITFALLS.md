# Domain Pitfalls

**Domain:** EdTech student aggregator dashboard — high school PII, role-based access, AI-generated academic planning
**Researched:** 2026-03-15
**Confidence:** MEDIUM — Based on Supabase/Next.js documentation patterns, FERPA regulation text, Gemini API capabilities, and direct analysis of the existing EduDashTool codebase. External web verification was unavailable during this session; all claims are grounded in official framework documentation or regulatory text, not opinion.

---

## Critical Pitfalls

Mistakes that cause rewrites, data breaches, FERPA violations, or complete loss of user trust.

---

### Pitfall 1: RLS Policies Written but Never Enforced at the Application Layer

**What goes wrong:** Developers write Supabase RLS policies and assume they automatically protect data accessed via the Drizzle ORM + direct `DATABASE_URL` Postgres connection. They do not. The `DATABASE_URL` connection in `src/db/index.ts` connects as the database owner (service role), which **bypasses all RLS policies entirely**. Every Drizzle query runs with superuser-equivalent access regardless of what RLS policies are configured in Supabase.

**Why it happens:** The Supabase dashboard gives the impression that enabling RLS on a table protects it globally. It does not — RLS only applies to connections through the Supabase PostgREST API or Supabase client SDK with the anon/JWT key. The raw `DATABASE_URL` Postgres connection used by Drizzle ORM runs outside that security boundary.

**Consequences:**
- A teacher querying `/api/students` can receive all school-wide student records if the API route doesn't manually filter by assigned classes
- The three-tier role model (teacher/counselor/principal) exists only in application code — a single missing `WHERE` clause exposes all student PII
- No Supabase RLS policy will catch this — the violation is invisible in RLS audit logs

**Prevention:**
- Every Drizzle query that returns student data MUST include explicit `WHERE` filters scoped to the authenticated user's role and assignments
- Build a `getStudentsForUser(userId, role)` data-access function that is the single path for all student queries — never query students directly in route handlers
- Add integration tests that verify a teacher JWT cannot retrieve students outside their class assignment
- Optionally: for the most sensitive queries (individual student profile), consider routing through Supabase client SDK (with anon key + JWT) so RLS applies as a second enforcement layer

**Detection (warning signs):**
- No `userId` or `role` parameter in a function that returns student records
- `db.select().from(students)` without a `.where()` clause anywhere in API routes
- Teacher role sees the same student count as counselor role

**Phase to address:** Schema and data access layer phase — must be correct before any student data can be entered.

### Pitfall 2: Student PII Sent to Gemini API Without FERPA Analysis

**What goes wrong:** Developers send raw student records — name, grade, attendance percentage, SAT score, GPA, behavioral notes — directly in Gemini API prompts to generate at-risk flags and roadmaps. This constitutes disclosing Personally Identifiable Information (PII) to a third-party vendor without the school having a valid FERPA exception or signed Data Processing Agreement (DPA) in place.

**Why it happens:** FERPA's "school official" exception allows disclosure to vendors with legitimate educational interest, but only if a DPA or equivalent agreement exists. Teams build the AI feature first and assume legal compliance is someone else's problem.

**Consequences:**
- School could be found in FERPA violation — penalties include loss of federal funding
- Google Gemini's default API terms do not constitute a FERPA-compliant DPA
- Student names + academic records in API logs or training data creates ongoing liability
- If the school is in California: SOPIPA and CCPA add additional restrictions on student data processed by EdTech vendors

**Prevention:**
- Before sending any student data to Gemini, confirm with the school that a Google Workspace for Education agreement or equivalent DPA is in place
- Implement a data minimization strategy: strip or pseudonymize PII before API calls where possible (send anonymized patterns, not named records: "Student A, 87% attendance, 2.4 GPA" instead of "John Smith, ...")
- Store Gemini prompts and responses in an audit log table with timestamps and which staff member triggered the AI call
- Add a clear UI disclosure that AI insights are generated by a third-party model and should not replace professional judgment
- Document which Google Gemini API endpoint is used and whether Google's data processing terms cover K-12 student data

**Detection (warning signs):**
- Gemini prompt construction includes `student.firstName`, `student.lastName`, or any direct identifier
- No audit log table exists for AI-generated content
- No mention of FERPA or DPA in project documentation

**Phase to address:** AI integration phase — before any live student data reaches the Gemini API.

### Pitfall 3: API Routes Not Checked for Authentication (Middleware Exclusion Gap)

**What goes wrong:** The existing middleware (`src/proxy.ts`) explicitly excludes `/api/*` routes from its matcher. This means every API route is responsible for calling `requireUser()` itself. If even one API route for student data is created without calling `requireUser()` first, it is publicly accessible without authentication.

**Why it happens:** The gap already exists in the codebase. New API routes added under `/api/students/`, `/api/ai/`, or `/api/analytics/` will silently be unprotected unless the developer explicitly adds the auth check. There is no framework-level enforcement.

**Consequences:**
- Unauthenticated HTTP requests to `/api/students` return student PII
- A FERPA breach that is trivially exploitable — no auth bypass needed, just direct URL access
- Would be invisible in Supabase auth logs because no auth check is ever called

**Prevention:**
- Create a shared API handler wrapper: `withAuth(handler)` that calls `requireUser()` and rejects before the handler executes — make this the only approved pattern for protected API routes
- Add a lint rule or code review checklist item: "Does this API route wrap with `withAuth`?"
- Write one automated test per student-facing API route that hits the endpoint without credentials and asserts `401`
- Consider adding the `/api/students/` and `/api/ai/` path prefixes back into the middleware matcher so unauthenticated requests are rejected at the edge before they reach route handlers

**Detection (warning signs):**
- API route file does not import from `src/lib/auth.ts`
- `fetch('/api/students/123')` returns data without an `Authorization` header in browser dev tools

**Phase to address:** API layer phase — must be resolved before any endpoint serves student records.

### Pitfall 4: Role Claims Trusted from Client-Side Without Server Verification

**What goes wrong:** The role system (teacher/counselor/principal) is implemented by storing role metadata in a database table or Supabase user metadata. On the client side, the role is read from session context and used to conditionally render UI. On the server side, API routes trust the role value passed in the request body or headers rather than re-fetching the authenticated user's role from the database.

**Why it happens:** It is tempting to pass `{ role: "counselor" }` in the API request payload to avoid an extra database lookup. This is a classic IDOR/privilege escalation pattern.

**Consequences:**
- A teacher changes their role claim to "principal" in the request payload and gains school-wide student access
- All student PII becomes accessible to any authenticated user regardless of role
- Role-based data filtering becomes purely decorative

**Prevention:**
- Never trust role information from the request body or query params
- Always derive the user's role by looking up `userId` (from the verified JWT) in the `staff_roles` table on the server side at the start of each protected request
- Build a `getAuthenticatedStaffContext(request)` function that returns `{ userId, role, assignedClassIds }` — use this as the source of truth in every API route, never the request payload
- Add a test: authenticated as teacher, send `role: "principal"` in request body, assert the response is scoped to teacher permissions only

**Detection (warning signs):**
- `req.body.role` or `searchParams.get('role')` used to determine data access scope
- Role not fetched from database in API route handlers

**Phase to address:** Role and permissions phase — foundational to all data access.

### Pitfall 5: Gemini AI Hallucinating Graduation Requirements and College Deadlines

**What goes wrong:** Gemini is prompted to generate a "college prep roadmap" and "graduation plan" for a student. The model hallucinates specific graduation credit requirements, SAT score cutoffs, college application deadlines, or financial aid dates that do not match the actual school's requirements or current admissions cycles. Staff act on these AI-generated plans without noticing the errors.

**Why it happens:** Gemini is a general-purpose language model. It has no knowledge of the specific school's graduation requirements, district policies, or current-year college deadlines. Without grounding in school-specific facts, it pattern-matches from training data and generates plausible-sounding but potentially incorrect specifics.

**Consequences:**
- A counselor forwards an AI-generated roadmap to a student showing wrong AP credit requirements
- A student misses a real application deadline because the AI generated an incorrect one
- Trust in the tool collapses after one visible error, killing adoption

**Prevention:**
- Never allow Gemini to invent school-specific facts: graduation credit requirements, district policies, and college-specific deadlines must be stored in the database and injected into the prompt as grounding context
- Prompts should follow the pattern: "Given these requirements: [db-fetched school requirements] and this student's data: [student record], generate a roadmap that fits within these constraints"
- Display a clear disclaimer on all AI-generated content: "AI-generated — verify against official school requirements before sharing with students or families"
- In the roadmap UI, clearly distinguish between facts sourced from the student's record (auditable) vs. AI suggestions (advisory)
- Build a "graduation requirements" configuration table in the database so school admins can specify their district's actual requirements — feed this into every Gemini prompt

**Detection (warning signs):**
- Gemini prompt does not include grounding data from the school's requirements table
- No disclaimer text rendered alongside AI-generated roadmap content
- AI response includes specific dates or credit numbers that aren't derived from the database

**Phase to address:** AI integration phase — prompt engineering must include constraint injection before going live.

### Pitfall 6: Supabase Anon Key Used in Client-Facing Code Without RLS Audit

**What goes wrong:** The `NEXT_PUBLIC_SUPABASE_ANON_KEY` is intentionally public, which is correct for Supabase's architecture. However, the Supabase tables have no RLS policies yet (schema is empty). When tables are created, if RLS is not enabled and policies are not written before the tables contain student data, the anon key can be used directly (e.g., via `curl` with the anon key) to query all student records from the PostgREST API.

**Why it happens:** Supabase tables are created with RLS disabled by default. The Drizzle ORM path works fine, so developers may not realize the Supabase REST API is also exposed.

**Prevention:**
- Enable RLS on every student-related table at creation time, before any data is inserted
- Add a Supabase policy that denies all access via the anon key to student tables (access should only go through authenticated server-side routes)
- In the Supabase dashboard, confirm the anon key cannot `SELECT` from `students`, `grades`, `attendance` tables
- Document this in the schema migration: "RLS must be enabled before running this migration in production"

**Detection (warning signs):**
- `curl https://[project].supabase.co/rest/v1/students -H "apikey: [anon-key]"` returns rows
- Supabase table settings show RLS as "disabled"

**Phase to address:** Database schema phase.

### Pitfall 7: Gemini API Costs Exploding on School-Wide Analytics

**What goes wrong:** The "school-wide trends" dashboard feature calls Gemini for every page load or every time a principal views the overview. With 500 students and detailed per-student data, each analytics call sends thousands of tokens. At scale this generates significant API costs that were not budgeted.

**Why it happens:** AI features feel lightweight during development with test data. Production load with real student populations reveals the cost structure.

**Prevention:**
- Cache Gemini responses: school-wide trend analysis does not need to regenerate on every page view — cache results with a TTL (e.g., 6-hour or 24-hour cache in the database)
- Store generated insights as a column in the database (`last_ai_analysis_at`, `ai_analysis_json`) and only re-call Gemini when the underlying data has changed or the cache TTL has expired
- Add a `GEMINI_API_KEY` budget alert in Google Cloud Console
- For individual student roadmaps: generate on-demand (button click) and cache the result until the student's data is updated — not on every profile view
- Estimate token costs before launch: school-wide analytics call size x principal daily usage x days

**Detection (warning signs):**
- Gemini API call triggered in a `useEffect` without a caching layer
- No `last_generated_at` timestamp on AI-generated content in the database

**Phase to address:** AI integration phase — caching strategy before any school-wide analytics feature.

### Pitfall 8: Manual Data Entry Errors Corrupting AI Insights

**What goes wrong:** Staff manually enter grades, attendance counts, and SAT scores. Typos (a grade of "110" instead of "11.0", or attendance of "10" instead of "100") are fed directly into Gemini prompts. The AI generates at-risk flags or roadmaps based on corrupted data. Staff notice wrong outputs but don't trace back to the bad input data.

**Why it happens:** Manual data entry without validation is inherently error-prone. The AI's confident-sounding outputs obscure bad inputs.

**Prevention:**
- Add strict Zod validation schemas for all student data entry forms with domain-specific constraints: GPA between 0.0–4.0, attendance percentage 0–100, SAT scores within College Board defined ranges (400–1600 combined)
- Show a data confidence indicator on AI-generated insights that links to the underlying data points used
- Build an audit log of who entered/modified each data point and when
- Add range-violation warnings (not just errors) for values that are technically valid but statistically unusual (e.g., "Attendance 15% — are you sure?")

**Detection (warning signs):**
- Student entry form has no `min`/`max` constraints or Zod refinements for academic fields
- No validation error shown when GPA of "5.0" is entered

**Phase to address:** Data entry forms phase.

### Pitfall 9: Teacher Scope Creep via Student Search

**What goes wrong:** The student list includes a search and filter feature. The backend filters students by teacher assignment when role is "teacher." However, the search endpoint is implemented as a simple `LIKE` query against all students, and the role filter is applied after retrieval (in JavaScript) rather than in the SQL `WHERE` clause. A teacher searching for a specific student name returns all matching students across the school, and the client-side filter is bypassed by inspecting the raw API response.

**Why it happens:** Search is implemented quickly as a database-wide text search. Role scoping is treated as a UI concern rather than a data access concern.

**Prevention:**
- All student search queries must JOIN against the teacher's class assignment table and filter in SQL — never in application code after the query returns
- The search API endpoint must accept only a `query` string, never a `studentId` list that can be crafted by the client
- Write a test: authenticated as teacher, search for a student not in their classes, assert zero results returned

**Detection (warning signs):**
- Search API route filters results using `.filter()` in JavaScript after DB query
- Search endpoint accepts `classId` override from client request

**Phase to address:** Student list and search phase.

### Pitfall 10: Missing Audit Trail for Student Data Access

**What goes wrong:** Staff can view and edit student PII with no record of who accessed which student record and when. If a FERPA complaint is filed ("Who looked at my child's records?"), there is no way to answer it. If a counselor's account is compromised, there is no way to determine what data was accessed.

**Why it happens:** Audit logging feels like overhead during MVP development and is deferred indefinitely.

**Consequences:**
- FERPA requires schools to maintain records of disclosures from student education records (34 CFR 99.32)
- Without access logs, a data breach investigation is impossible
- Staff cannot be held accountable for misuse of sensitive data

**Prevention:**
- Build a `record_access_log` table: `(id, staff_user_id, student_id, action, accessed_at, ip_address)`
- Log every view of a student profile page and every edit of student data — this can be lightweight (one DB insert per access)
- Log every Gemini API call that includes student data: who triggered it, for which student, and store the AI response hash
- This table should be append-only: no staff role should have DELETE permission on the access log

**Detection (warning signs):**
- No `record_access_log` or equivalent audit table in the schema
- No logging call when a student profile is loaded in the server component or API route

**Phase to address:** Core data model phase — access logging table created alongside student tables.

### Pitfall 11: Silent Auth Failures Masking Real Errors (Existing Codebase Issue)

**What goes wrong:** The existing `src/lib/supabase/client.ts` uses placeholder fallback URLs; `src/lib/supabase/server.ts` uses empty string defaults. When environment variables are missing in a new deployment, auth fails silently — the app loads but cannot authenticate anyone.

**Prevention:**
- Add startup validation that throws a descriptive error if `NEXT_PUBLIC_SUPABASE_URL` or `NEXT_PUBLIC_SUPABASE_ANON_KEY` are empty or contain placeholder values
- The health check endpoint should also verify Supabase connectivity, not just the database

**Detection:** Login page appears but all auth attempts fail with cryptic errors; no explicit startup error shown.

**Phase to address:** Rebrand/setup phase — fix before building any new features.

---

### Pitfall 12: Drizzle Build-Time Migrations Running Against Production Database

**What goes wrong:** The build script is `"build": "drizzle-kit migrate && next build"`. This means every Vercel deployment runs database migrations automatically. If a migration has an error or introduces a breaking schema change, the production database is affected before the deployment can be rolled back.

**Prevention:**
- Separate migration step from build step: run migrations as a separate deploy step with explicit confirmation, not as part of `next build`
- Use Vercel's deploy hooks or a pre-deploy script rather than embedding migrations in the build command
- Always test migrations against a staging database before production

**Detection:** Schema migration error causes build to fail mid-way; partial migrations leave database in inconsistent state.

**Phase to address:** Schema design phase — before any production data exists.

---

### Pitfall 13: Proxy Middleware Redirecting to `/songs` (Existing Dead Route)

**What goes wrong:** `src/proxy.ts` redirects authenticated users to `/songs`, which does not exist and will not exist in the rebranded EduDash app. This means every fresh login results in a 404.

**Prevention:**
- Update all redirect targets in `src/proxy.ts` when rebranding: `/songs` → `/dashboard` or `/students`
- Extract route constants to a single file to prevent this class of error in future

**Detection:** Login succeeds, then user lands on a 404 page.

**Phase to address:** Rebrand phase — immediate fix before any new development.

## Phase-Specific Warnings

| Phase Topic | Likely Pitfall | Mitigation |
|-------------|---------------|------------|
| Rebrand / setup | Dead `/songs` redirect breaks post-login flow | Fix `proxy.ts` redirect targets first |
| Database schema design | RLS disabled on student tables by default | Enable RLS at table creation; add deny-all anon policy |
| Role system implementation | Role claims trusted from client payload | Always derive role from DB lookup in server; never from request body |
| Data entry forms | No input validation allows corrupted academic data | Zod schemas with domain constraints on all numeric academic fields |
| Student list and search | Search bypasses teacher scope filter | SQL-layer role filter on every student query; test with teacher JWT |
| AI integration | Raw student PII in Gemini prompts (FERPA) | Pseudonymize where possible; confirm DPA; add audit log |
| AI integration | Gemini invents graduation requirements | Inject school requirement data from DB as grounding context |
| AI integration | Unbounded API costs on school-wide analytics | Cache AI results in DB; only re-generate when source data changes |
| API routes | Missing `requireUser()` calls on new routes | `withAuth()` wrapper pattern; test unauthenticated access returns 401 |
| Access logging | No FERPA-required disclosure record | Build audit log table alongside student tables in same phase |
